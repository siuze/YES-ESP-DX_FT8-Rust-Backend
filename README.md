# YE'S-ESP-DX_FT8-Rust-Backend

一个用于控制本人自制短波电台YE'S-ESP-DX的后端服务程序。

## 功能列表
在AI辅助编程下，几乎使用纯Rust实现，已实现的功能与计划如下：

- [ ] 自动获取电台IP并设置上位机IP
- [x] 持续监听电台的IQ采样数据报文
- [x] 持续监听并转发电台日志报文到前端
- [x] IQ采样数据解调（生成信号瀑布图并发送到前端）
- [x] FT8信号解码（消息文本，频偏，时偏，SNR，网格距离，呼号归属地区）
- [ ] FT8消息编码为音频符号（目前通过调用WSJTX-lib C库实现）
- [x] 按需下发FT8发射命令到电台
- [x] 单次或持续处理FT8解码结果并回复发射或主动CQ（自动优选频率窗口）
- [x] FT8消息解码结果上报 PSK reporter
- [x] 通联日志记录到 Notion数据库
- [ ] 解调并传输USB/LSB/AM音频到前端
- [ ] 解码短波气象传真消息
- [ ] 测试模式：带通+低通接收滤波性能扫描
- [ ] 测试模式：镜像抑制比/边带抑制比扫描
- [ ] 测试模式：IQ正交平衡幅度/相位检测
- [ ] 测试模式：FT8音频生成（高斯滤波）效果测试
- [ ] 测试模式：天线驻波比扫描




## FT8解码
- 我希望能够在ARM嵌入式设备上运行更好的FT8解码算法，以捕获微弱信号，因此不直接使用ft8_clib;
- FT8信号解码参考了WSJTX-Lib项目代码，目前除了AP解码外，理论上其余解码算法已经完全移植到了Rust，解码性能基本表现一致。

## 现存问题：

- 有时会解码出错误消息（非常长且杂乱的呼号）；
- 哈希呼号解码似乎仍有问题；
- 在RK3566上运行，解码15s音频数据的总耗时仍然较长，不够理想；
- SNR结果与WSJTX-Lib虽然在量级上相差不大，但仍有差异

## 有关前后端

为YE'S-ESP-DX的创建独立的后端服务有很多原因：

- 我人常在外地，与家中设备的互联常不通畅，为了让采样信号的处理与消息收发尽可能靠近电台侧，减少网络延迟与波动，需要在家中局域网部署一个能够自运行的后台进程。
- 通过前后端分离，我只需要通过PC侧的前端APP偶尔查看一下运行状态，消息日志和操纵一下运行模式即可，所有的持续运行逻辑都放在后端进行；
